## 1. 项目概述

### 1.1 背景
XX市安全生产责任保险信息云平台（安责险平台）是面向安责险全业务链条的数字化管理系统，覆盖投保、保全、理赔、预防服务、政府监管、财务结算等核心业务。3.0版本在2.x版本基础上，重构前端架构，引入AI能力，统一多端（PC管理端 + 小程序端）交互体验。

### 1.2 目标
- **可用性**：支持5类角色端口，满足政府监管、保险公司、服务机构、运营平台、企业用户的差异化需求
- **AI能力**：OCR保单识别、AI理赔识别、AI行业画像、AI报表生成、AI智能客服
- **数据闭环**：从投保 → 保单管理 → 预防服务 → 理赔 → 结算 全流程数字化
- **监管合规**：支持数据上报、数据质控、数据直报，满足上级平台对接需求

### 1.3 架构形式
- **PC端**：`index.html`原型，
- **小程序端**：`xcx.html`原型
- **后端**：RESTful API（本文档不含后端接口规范，需另行出接口文档）

## 3. 用户角色与端口

系统共5个端口（Portal）

| 端口 | data-portal | 角色描述 | 特有功能 |
|------|-------------|----------|---------|
| 运营管理员端 | `operation` | 平台运营方，拥有全部权限 | 企业匹配、AI报表中心、业务员业绩统计、系统运维、结算规则配置 |
| 政府端 | `government` | 政府安监部门，只读+审批 | 任务检查（发起方）、数据上报 |
| 保险业务员端 | `insurance` | 保险公司业务员 | 批单管理、OCR识别、费用结算 |
| 服务专家端 | `service` | 安全服务机构专家 | 集中服务列表、企业自主排查（协助） |
| 企业用户端 | `enterprise` | 投保企业用户 | 企业自主排查（发起）、微信绑定 |

## 4. 全局交互规范

### 4.1 侧边栏交互
- **展开宽度**：240px，**收起宽度**：64px（仅显示icon）
- 顶部搜索框：输入关键词实时过滤菜单项，匹配高亮 `.highlight`，不匹配追加 `.hidden-by-search`
- 二级菜单：点击父菜单展开/收起子菜单（Accordion式，同一时间只展开一个父菜单）
- 菜单角标：红色圆形badge显示待处理数量
- 底部常驻：切换主题 / 切换端口 / 退出登录

### 4.2 顶部Header
- 高度：56px，背景：`rgba(10,22,40,0.95)`，底部边框：`rgba(0,212,255,0.2)`
- 左侧：面包屑导航（`/`分隔，当前页加粗白色）
- 右侧：通知bell（红色badge数字）+ 用户头像 + 用户名 + 下拉箭头
- 下拉菜单：个人设置 / 切换端口 / 退出登录

### 4.3 通知系统
- 铃铛图标点击展开通知面板（Dropdown）
- 通知类型：系统通知 / 业务提醒 / 待办事项
- 已读/未读状态，支持"全部已读"批量操作
- 未读总数显示在badge

### 4.4 弹窗规范
- **Modal弹窗**：宽度480px~800px不等，居中显示，含mask蒙层
- **Drawer抽屉**：从右侧滑入，宽度800px~1000px，不遮挡侧边栏
- 关闭方式：右上角×按钮 / 点击mask / ESC键（抽屉不响应mask点击关闭）
- 表单弹窗底部：取消按钮（灰色）+ 确认按钮（蓝色主色）
- 危险操作弹窗底部：取消按钮 + 确认删除按钮（红色 `#ff4d4f`）

### 4.5 表格通用规范
- 表头背景：`#fafafa`，字体颜色：`#666666`
- 奇偶行不区分颜色（统一白色）
- Hover行：`#f5f5f5`
- 选中行：`#e6f7ff`（蓝色浅底）
- 固定列：checkbox列固定左侧（z-index:5），操作列固定右侧（左侧有box-shadow）
- 超长文字：`overflow:hidden; text-overflow:ellipsis; white-space:nowrap`，hover展开tooltip
- 数字列右对齐（`.col-number`）
- 分页：左侧显示"共X条"，右侧分页按钮，支持切换每页条数（10/20/50/100）

### 4.6 状态标签（Tag）规范
| 样式类 | 颜色 | 适用状态 |

### 4.7 按钮规范
| 按钮类型 | 样式 | 适用场景 |
|---------|------|---------|
| 主要按钮 `.btn-primary` | 蓝色实心 | 新增、确认、提交 |

## 5. 通用组件规范

### 5.1 筛选栏组件
- 位于页面顶部，白色卡片内
- 布局：行内flex，每项标签+控件，右侧"查询"主按钮+"重置"默认按钮
- 控件类型：文本输入框 / 下拉选择框 / 日期范围选择器 / 级联选择器
- 查询触发：点击查询按钮 或 按Enter键
- 重置：清空所有筛选条件，重新加载全量数据

### 5.2 统计卡片组件（`.pc-stat-card`）
- 宽度：1/4等分（grid 4列）
- 高度：约100px
- 内容：顶部小标签（13px）+ 大数字（28px bold）+ 底部趋势说明（12px）
- 颜色：蓝色/绿色/橙色/紫色渐变背景，白色文字

### 5.3 数据表格+分页组件
- 表格需支持：列显示/隐藏切换（右上角icon）、列宽拖拽（可选）
- 分页默认每页20条
- 批量操作：勾选后顶部出现批量操作浮层（含已选X条文字 + 操作按钮）

### 5.4 上传组件
- 支持拖拽上传 + 点击上传
- 格式限制显示在上传区域下方（红色小字）
- 上传后显示文件名+大小+删除按钮
- 上传进度条（蓝色）

### 5.5 Rich文本/预览组件
- 用于服务报告、排查记录等内容展示
- 支持图片预览放大

## 8. 数据字典

### 8.1 枚举值定义

**保单状态**：
- `PENDING`：待起保
- `ACTIVE`：在保
- `EXPIRING`：临期（30天内到期）
- `EXPIRED`：已脱保
- `VOID`：已失效
- `CANCELLED`：已退保

**服务状态**：
- `NOT_STARTED`：未开始
- `IN_PROGRESS`：进行中
- `REVIEWING`：待审核
- `COMPLETED`：已完成
- `REJECTED`：已驳回

**理赔状态**：
- `REPORTED`：已报案
- `ASSESSING`：定损中
- `PROCESSING`：理赔中
- `SETTLED`：已结案
- `WITHDRAWN`：已撤案

**AI监测结果**：
- `NORMAL`：正常
- `WARNING`：预警
- `ABNORMAL`：异常

**任务状态**：
- `PENDING_ASSIGN`：待分配
- `IN_PROGRESS`：执行中
- `PENDING_REVIEW`：待审核
- `COMPLETED`：已完成
- `OVERDUE`：已逾期
- `REJECTED`：已驳回

**签章状态**：
- `PENDING`：待审核
- `APPROVED`：已批准
- `REJECTED`：已拒绝
- `WITHDRAWN`：已撤回

**账单状态**：
- `DRAFT`：草稿
- `PENDING_CONFIRM`：待确认
- `CONFIRMED`：已确认
- `DISPUTED`：争议中
- `SETTLED`：已结算

**数据上报校验状态**：
- `PENDING`：待上报
- `SUCCESS`：上传成功
- `FAILED`：上传失败
- `PARTIAL`：部分成功

**隐患等级**：
- `MAJOR`：重大（红色标注）
- `GENERAL`：一般（橙色标注）
- `MINOR`：轻微（黄色标注）

**日志级别**：
- `INFO`：信息（蓝色）
- `WARN`：警告（橙色）
- `ERROR`：错误（红色）

### 8.2 状态Tag颜色映射
所有模块状态颜色遵循统一规范：
- 绿色（green tag）→ 成功、正常、完成类状态

- 红色（red tag）→ 失败、异常、紧急类状态

### 10.1 数据统计模块 — 字段来源与计算规则

#### 10.1.1 Dashboard 首页统计卡片

| 指标 | 数据来源 | 计算规则 |
|------|---------|---------|
| 在保保单（件） | `policy_list` 表 | `COUNT(*) WHERE policy_status = 'ACTIVE'` |
| 脱保预警（件） | `policy_list` 表 | `COUNT(*) WHERE policy_end_date BETWEEN NOW() AND DATE_ADD(NOW(), INTERVAL 30 DAY) AND policy_status = 'ACTIVE'` |
| 本月理赔案件（件） | `claim_list` 表 | `COUNT(*) WHERE created_time BETWEEN 当月1日 AND 当月最后一日` |
| 服务覆盖率（%） | `policy_list` + `prevent_service` 表 | `COUNT(DISTINCT enterprise_id FROM prevent_service WHERE status='COMPLETED') / COUNT(DISTINCT enterprise_id FROM policy_list WHERE status='ACTIVE') × 100` |
| 参保企业数（家） | `policy_list` 表 | `COUNT(DISTINCT enterprise_id) WHERE policy_status = 'ACTIVE'` |

**月度对比**：每张卡片底部 "较上月 +32 ↑" 通过比较本月与上月数据差值计算：
```
变化率 = (本月值 - 上月值) / 上月值 × 100
正数显示绿色 ↑，负数显示红色 ↓
```

#### 10.1.2 简要统计 — 总览卡片

| 指标 | 数据来源 | 计算规则 |
|------|---------|---------|
| 应投保企业/项目数 | `gov_register` 表（政府备案数据） | `COUNT(*) WHERE status != 'CANCELLED'` |
| 在保企业/项目数 | `policy_list` 表 | `COUNT(DISTINCT enterprise_id) WHERE policy_status = 'ACTIVE'` |
| 保费规模 | `policy_list` 表 | `SUM(premium) WHERE policy_status IN ('ACTIVE','EXPIRED')` |
| 赔付金额 | `claim_list` 表 | `SUM(settlement_amount) WHERE status = 'SETTLED'` |
| 赔付家数 | `claim_list` 表 | `COUNT(DISTINCT enterprise_id) WHERE status = 'SETTLED'` |
| 赔付次数 | `claim_list` 表 | `COUNT(*) WHERE status = 'SETTLED'` |
| 服务开展家数 | `prevent_service` 表 | `COUNT(DISTINCT enterprise_id) WHERE status IN ('COMPLETED','IN_PROGRESS')` |
| 服务费用累计 | `prevent_service` 表 | `SUM(service_fee) WHERE status != 'NOT_STARTED'` |

#### 10.1.3 政府部门统计（Tab1）

**数据来源**：从 `policy_list` + `gov_register` + `prevent_service` + `claim_list` 四表联合聚合，按 `视角切换` 字段分组。

**视角切换规则**：
- **按行政区域**：`GROUP BY area`，分组字段取自 `enterprise.area`
- **按行业**：`GROUP BY industry`，分组字段取自 `enterprise.industry`
- **按保险机构**：`GROUP BY insurance_org_id`，分组字段取自 `policy_list.insurance_org_id`

**核心计算公式**：
```
投保率 = 已投保 / 应投保 × 100
服务开展率 = 服务开展家数 / 已投保企业数 × 100
投保率 = 投保企业数 / 应投保企业数 × 100 (%)  -- 假设计算为 insured / shouldInsure
```

**汇总行（合计行）**：对筛选后数据所有数值列求和，投保率=总已投保/总应投保×100，服务开展率=总服务家数/总已投保×100。

#### 10.1.4 保险业务统计（Tab2）

**数据来源**：`policy_list` + `prevent_service` + `claim_list`，按 `保险机构` 分组。

**本地/外地分类** (`type: 'local' | 'out'`)：
- local：保险机构注册地与保单所属地一致
- out：保险机构注册地与保单所属地不一致（外地承保）

**汇总行**：参保企业、保费规模、保障额度、导入保单数量、服务开展家数、服务开展次数、服务累计金额、赔付家数、赔付次数、赔付金额全部求和。

#### 10.1.5 事故预防服务统计（Tab3）

**数据来源**：`prevent_service` + `policy_list`，按 `服务项目` 分组。

**核心字段计算公式**：
```
服务开展率 = 服务开展家数 / 投保企业家数 × 100 (%)
列支费用占比 = 服务费用列支金额 / 应提取费用金额 × 100 (%)
服务执行率 = 实际执行的服务费用 / 应提取费用金额 × 100 (%)
剩余服务费用 = 应提取费用金额 - 已列支费用金额
```

**应提取费用金额**：根据结算规则配置中的提取比例计算（见 10.7 节）。

#### 10.1.6 AI隐患识别统计（Tab4）

**数据来源**：`ai_hazard_detection` 表。

**核心计算**：
```
整改率 = 已整改 / 隐患总数 × 100 (%)
```

**AI分析概要文字**：在每个筛选条件变更后，自动生成文字描述，格式如：
`"基于{行业}行业在{地区}的{企业数}家企业数据分析，共识别隐患{总数}项，整体整改率为{整改率}%。其中{待整改最多的行业}行业待整改数量最高，需重点关注。"`

### 10.2 保单生命周期 — 状态流转规则

#### 10.2.1 保单状态机

```
┌─────────┐    到达起保日期     ┌─────────┐    距到期≤30天    ┌──────────┐
│ PENDING │ ─────────────────► │ ACTIVE  │ ───────────────► │ EXPIRING │
│ 待起保  │                    │  在保   │                  │  临期    │
└─────────┘                    └─────────┘                  └──────────┘
     │                              │                            │
     │ 手动退保                     │ 保单到期日到达              │ 保单到期日到达
     ▼                              ▼                            ▼
┌───────────┐                ┌──────────┐                 ┌──────────┐
│ CANCELLED │                │ EXPIRED  │ ◄───────────────│ EXPIRED  │
│ 已退保    │                │  已脱保  │                 │  已脱保  │
└───────────┘                └──────────┘                 └──────────┘
                                   ▲
┌─────────┐                       │ 手动标记失效
│  VOID   │ ──────────────────────┘
│ 已失效  │
└─────────┘
```

#### 10.2.2 状态触发条件

| 状态 | 触发条件 | 触发方式 | 频率 |
|------|---------|---------|------|
| `PENDING` → `ACTIVE` | 系统时间 ≥ 保单起始日期 | 系统定时任务自动 | 每日凌晨 |
| `ACTIVE` → `EXPIRING` | 保单终止日期 - 当前日期 ≤ 30天 | 系统定时任务自动 | 每日凌晨 |
| `ACTIVE`/`EXPIRING` → `EXPIRED` | 保单终止日期 < 当前日期 | 系统定时任务自动 | 每日凌晨 |
| 任意 → `CANCELLED` | 用户手动退保操作 | 人工触发（需二次确认） | 按需 |
| 任意 → `VOID` | 运营端/保险端手动标记失效 | 人工触发 | 按需 |

**定时任务伪代码**：
```
function dailyPolicyStatusUpdate():
    today = NOW()
    // 待起保 -> 在保
    UPDATE policies SET status='ACTIVE'
    WHERE status='PENDING' AND start_date <= today
    
    // 在保 -> 临期
    UPDATE policies SET status='EXPIRING'
    WHERE status='ACTIVE' AND end_date - today <= 30 AND end_date > today
    
    // 在保/临期 -> 脱保
    UPDATE policies SET status='EXPIRED'
    WHERE status IN ('ACTIVE','EXPIRING') AND end_date < today
```

#### 10.2.3 续保逻辑

**触发条件**：
- 保单 `status IN ('EXPIRING','EXPIRED')` 时，保单列表操作列显示"续保"按钮
- 续保弹窗中：自动填充原保单的企业、保险机构等信息（只读）
- 用户填写新保险期间 + 可选调整保费
- 确认后：生成新保单记录（新保单号），状态为 `PENDING`

**数据流转**：
```
原保单(EXPIRED) + 续保操作 → INSERT新保单(PENDING)
新保单.premium = 原保单.premium（除非用户手动调整）
新保单.policy_no = 新生成保单号
关联关系：新保单.renewed_from = 原保单.id
```

| 状态变更 | 触发条件 | 触发操作 | 执行后效果 |
|---------|---------|---------|-----------|
| 新建 → `待分配` | 运营/政府端创建任务，未指定执行机构 | `createTask()` | 进度=0，handler='-' |

### 10.6 账单/费用结算 — 状态流转与计算逻辑

#### 10.6.1 费用结算字段数据来源

**预防服务费用结算表**（22列）的数据聚合逻辑：

| 字段 | 数据来源 | 说明 |
|------|---------|------|
| 保单号 | `policy_list.policy_no` | 关联保单 |
| 企业名称 | `enterprise.name` | 关联企业 |
| 保费(元) | `policy_list.premium` | 保单保费 |
| 计提比例(%) | `settlement_rules.extract_ratio` | 从结算规则读取 |
| 计提金额(元) | `premium × extract_ratio / 100` | **自动计算**：保费 × 计提比例 |
| 已服务次数 | `COUNT(prevent_service WHERE status IN ('COMPLETED','IN_PROGRESS'))` | 关联服务记录 |
| 已服务费用(元) | `SUM(prevent_service.service_fee)` | 已开展服务的费用合计 |
| 费用余额(元) | `计提金额 - 已服务费用` | **自动计算** |

**关键公式**：
```
计提金额 = 保费(元) × 计提比例(%) / 100
费用余额 = 计提金额 - 已服务费用
应提取费用金额 = 保费 × 规则中配置的比例
```

#### 10.6.2 保险机构账单状态流转

```

│  DRAFT │ ──────────► │ PENDING_  │ ──────────► │CONFIRMED │ ──────────► │ SETTLED  │
│  草稿  │             │ CONFIRM   │              │  已确认   │            │  已结算   │
└────────┘             │  待确认    │              └──────────┘            └──────────┘
                        └───────────┘                   │
                              │                         │ 争议
                              │                         ▼
                              │                   ┌──────────┐
                              │                   │ DISPUTED │
                              │                   │  争议中   │
                              │                   └──────────┘
```

#### 10.6.3 账单操作触发条件

| 操作 | 触发条件 | 执行后状态 |
|------|---------|-----------|
| 保存草稿 (`feeSaveDraft`) | 账单状态为 `DRAFT` | 仍为 `DRAFT`，保存当前勾选和数据 |
| 推送账单 | 账单状态为 `DRAFT`，至少勾选1条保单 | 发送通知给保险机构，状态→`PENDING_CONFIRM` |

#### 10.6.4 机构间对账逻辑

**数据一致性判断**：
```javascript
if (保险方确认金额 === 服务方确认金额) → '一致'（绿色✓）
else if (|保险方确认金额 - 服务方确认金额| ≤ 容差范围) → '存疑'（橙色!）
else → '争议'（红色✗）
```

**争议沟通室**（抽屉 800px 宽）：
- 金额对比表：保险方金额 vs 服务方金额 vs 差异
- 聊天记录区：时间线展示双方沟通
- 输入框：发送消息 + 可选附件
- 仲裁按钮：仅运营管理员可见，点击后弹出确认框，锁定最终金额

#### 10.6.5 结算规则 — 费率计算公式

**监测规则配置模块中的费率计算器** (`calcMrcRate`)：

```
最终费率 = 基础费率 × 工期系数 × 造价系数 × 类型系数
```

**系数表**：

| 因子 | 区间 | 系数值 |
|------|------|--------|
| **工期系数** | ≤12个月 | 0.9 |
| | 13-36个月 | 1.0 |
| | 37-60个月 | 1.2 |
| | >60个月 | 需议价（标记为-1） |
| **造价系数** | <3000万 | 1.3 |
| | 3000-5000万 | 1.2 |
| | 5000-10000万 | 1.1 |
| | 10000-50000万 | 1.0 |
| | 50000-100000万 | 0.9 |
| | >100000万 | 需议价（标记为-1） |

**异常处理**：工期系数或造价系数为-1时，显示"需议价"，不自动计算，需人工逐单定价。
